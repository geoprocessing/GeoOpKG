# GeoOpKG: Import into Neo4j

This directory (`Import into Neo4j`) contains the source data files (CSV) and data import scripts (Python) used to build the **Geospatial Operator Knowledge Graph (GeoOpKG)**. Through these scripts, you can transform various geospatial operations (such as operators in ArcGIS, QGIS, GEE, and other software) into nodes and relationships, and import them into a Neo4j database.

## 1. Directory Structure and Purpose

### 1.1 Data Files (CSV)
These tables are data extracted or cleaned from original software tools, serving as raw materials for building the knowledge graph:
- **`[Software]_Input.csv`**: (e.g., `ArcGIS_Input.csv`) Records the specific operators (Operation) of the software and their required **input parameter information**. The data format typically includes: software name, software version, operator name, operator ID, operator description, and variable-length sequences of (input parameter name, input description, etc.).
- **`[Software]_Output.csv`**: (e.g., `ArcGIS_Output.csv`) Records the **output parameter information** (including output name and description) generated after the execution of the corresponding operator (ID).
- **`category.csv`**: Contains the top-level defined **abstract algorithm concept layer (Algorithm)** and hierarchical classification of the graph.
- **`[Software]_Algorithm.csv`**: (e.g., `ArcGIS_Algorithm.csv`) Algorithm mapping table data (generated by LLMs or interactive classification), used to specify the general category "Algorithm" corresponding to a certain operator ID.

### 1.2 Import Scripts (Python)
These scripts use libraries like `pandas` and `py2neo` / `neo4j` to connect to the local database, parse row structures and dependency relationships in CSVs one by one, and use Cypher commands to create Nodes and Relationships.
- **`import_[Software]_Input.py`**: Imports the operator itself, its development software, and its input requirements into the database. It generates nodes like `Operation`, `Software`, and `Input`, and creates relationships like `ImplementedIn` and `hasInput`.
- **`import_[Software]_Output.py`**: Based on the operator ID, imports the product (output result) information into the database. It generates output nodes and creates `hasOutput` relationships with the `Operation` node.
- **`import_category.py`**: Loads the specific abstract algorithm hierarchies (Algorithm nodes) into the database, used to construct multi-dimensional classification capabilities.
- **`import_operator_algorithm_relation.py`**: Reads `*_Algorithm.csv` to connect specific operators with abstract algorithm concepts (creates `hasPlan` relationships).

## 2. Complete Neo4j Build Method

To smoothly import all operator data and their associations from scratch into a local Neo4j instance, please follow the steps below.

### Prerequisites

1. **Environment Preparation**: 
   Ensure Python dependencies are installed:
   ```bash
   pip install pandas py2neo neo4j
   ```
2. **Start Neo4j Service**: 
   Create a database in Neo4j Desktop or the server application. Ensure it is accessible via `bolt://localhost:7687`.
3. **Check and Verify Configuration**:
   Open the four types of scripts and check the password configuration at `Graph("bolt://localhost:7687", auth=("neo4j", "your_password"))` or `GraphDatabase.driver`. If no password is set locally, leave it as an empty string; otherwise, please modify it to the real `neo4j` account password.

### Execution Steps

It is recommended to run the following code in order from primary to secondary according to the entity dependency hierarchy to avoid relationship construction failures caused by missing referenced nodes:

#### **Step 1: Import Concept Classification and Algorithm Definitions**
Run the category script to lay the foundation for abstract classification in the database first (this part is independent of specific software):
```bash
python import_category.py
```

#### **Step 2: Import Software Operators and Their Inputs (Input)**
Run the corresponding input script for each GIS software framework. This will create software, operators, and link them:
```bash
python import_ArcGIS_Input.py
python import_QGIS_Input.py
python import_GEE_Input.py
# Run all provided Input scripts in a similar manner
```

#### **Step 3: Append and Bind Operator Outputs (Output)**
Based on the operator ID, bind the product (output result) information to the operators created previously:
```bash
python import_ArcGIS_Output.py
python import_QGIS_Output.py
python import_GEE_Output.py
# Run all provided Output scripts in a similar manner
```

#### **Step 4: Match Operators with Algorithm Classification Models**
Finally, attach the implementation capabilities of operators to the top-level algorithm network. Since the data tables of different software are often separate, you need to read the corresponding `*_Algorithm.csv` files one by one to execute the binding.
(Note: If necessary, you might need to slightly adjust the file path parameter inside `import_operator_algorithm_relation.py` so it can cover all software tables, e.g.: first change it to `ArcGIS_Algorithm.csv` and run once, then change it to `QGIS_Algorithm.csv` and run once...)
```bash
python import_operator_algorithm_relation.py
```

**✅ Completion Check**
At the end of this step, your GeoOpKG operator knowledge graph has been successfully loaded into local Neo4j. You can switch back to the Neo4j Browser interface to query:
```cypher
// Test query an operator and all its relationships
MATCH (o:Operation)-[r]-(n) RETURN o, r, n LIMIT 25;
```
